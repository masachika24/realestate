## 1. プロジェクト概要

### 1.1 目的・背景

**realestate** は、AI エージェント（Opencode）による**不動産投資業務の自動化・効率化**を目的とした事業プロジェクトである。

本プロジェクトは **Executive プラットフォーム**（`C:\dev\executive`）上に構築され、全事業共通の基盤機能（タスク管理、品質保証、セッション管理）を活用しつつ、不動産投資ドメインに特化したエージェント群を段階的に導入する。

### 1.2 Executive プラットフォームとの関係

| 層 | 役割 | 実体 |
|----|------|------|
| Executive（グローバル） | 基盤インフラ、全事業共通ツール、品質保証、セッション管理 | dashboard_* ツール、handoff、orchestrator、quality-gate、dispatcher |
| realestate（事業ローカル） | 不動産投資ドメイン知識、業務特化エージェント、投資判断ロジック | property-scout、deal-analyzer、contract-reviewer 等 |

> **運用原則**: ユーザーは dispatcher とのみ対話する。orchestrator は内部的にタスクを分解・統合し、事業特化エージェントはプロジェクトローカルで稼働する。

## 2. 投資戦略

### 2.1 投資方針

| 項目 | 内容 |
|------|------|
| 主目的 | 減価償却による税務メリットの最大化 |
| 保有期間 | 7〜15年の中期保有 |
| 収益要件 | キャッシュフロー黒字（ローン返済後） |
| 出口戦略 | 売却時に購入価格から大きく下落しない物件を選定 |

### 2.2 既存ポートフォリオ

| 項目 | 内容 |
|------|------|
| 保有物件 | 札幌市に3棟保有 |
| 投資家所在地 | 東京都在住 |
| 運用形態 | 遠隔運用中（管理会社委託） |

## 3. 対象エリア

### 3.1 エリア一覧

| エリア | 選定理論根拠 | 備考 |
|--------|-------------|------|
| 札幌市 | 既存取引実績・管理基盤あり。3棟の運用実績から地域特性・業者ネットワークを把握済み | 優先エリア |
| 関東圏（埼玉・千葉・神奈川の一部） | 東京から日帰りアクセス可能。現地確認のハードルが低く、管理会社への依存度を低減可能 | 拡大候補エリア |

### 3.2 エリア選定基準

- 東京から公共交通機関または車で日帰り往復可能な距離
- 人口動態が安定または微増傾向
- 再開発計画や大規模インフラ整備が見込めるエリア

## 4. 物件探索チャネルと AI 活用方針

### 4.1 ポータルサイト

| 項目 | 内容 |
|------|------|
| 媒体 | athome、SUUMO、楽待、健美家、Yahoo! 不動産 |
| 強み | 条件合致のメール通知機能により、新着物件を能動的にキャッチ可能 |
| 課題 | 優良物件の競争が激化。利回り良好物件は掲載後数時間で消える。通知→即判断の仕組みが必須 |
| AI 方針 | **property-scout** で通知メールを自動解析し **deal-analyzer** で即時収支計算。閾値超過物件を **dispatcher** 経由でユーザーに即時通知 |

### 4.2 業者ネットワーク

| 項目 | 内容 |
|------|------|
| 現状 | 地場優良物件はポータル掲載前に業者間で回る。業者との関係構築が物件取得の鍵 |
| 課題 | 人間関係に依存。東京在住で札幌業者との関係維持が物理的に困難 |
| AI 方針 | **broker-liaison** で定期コンタクトを自動化（人間の関与は最終段階のみ）。関係維持の工数を最小化 |

### 4.3 競売・公売

| 項目 | 内容 |
|------|------|
| 情報源 | 裁判所 BIT（競売情報）、自治体公売情報 |
| 課題 | 3点セット（物件明細書・現況調査報告書・評価書）の読み込みが必要。占有者問題などの法的リスクが内在 |
| AI 方針 | **off-market-finder** で定期的に巡回し、**deal-analyzer** で粗利計算。法的リスクは **contract-reviewer** で一次判定 |

### 4.4 オフマーケット・ダイレクト

| 項目 | 内容 |
|------|------|
| 手法 | 空き家所有者への DM 送付、相続物件の登記異動監視 |
| 課題 | 発掘に時間がかかる。レスポンス率が低く、ROI の見極めが必要 |
| AI 方針 | 長期的に **off-market-finder** が定期巡回。データ蓄積により精度向上を図る |

## 5. エージェント導入計画

`C:\dev\realestate\docs\agent-blueprint.md` の優先度マップに基づき、以下の4段階で導入を計画する。

| Phase | エージェント | 目的 | 期待効果 |
|-------|-------------|------|---------|
| Phase 1 | **property-scout**、**deal-analyzer** | 物件発見〜収支計算の自動化。機会損失防止 | 日次業務時間 20〜30% 削減、投資判断スピード 2〜3 倍 |
| Phase 2 | **area-analyst**、**market-researcher** | エリアの将来性・価格動向の定量評価 | 情報収集時間 50% 削減、契約リスク見落とし率 90% 低減 |
| Phase 3 | **email-reader**、**email-drafter**、**contract-reviewer** | コミュニケーションの効率化・契約書チェック自動化 | 運用管理工数 40〜50% 削減、滞納リスク早期検知率 80% 向上 |
| Phase 4 | 運用系（**lease-manager** 等）＋意思決定支援系（**portfolio-visualizer** 等） | 保有物件の運用効率化＋ポートフォリオ最適化 | 全業務領域の AI カバレッジ 80% 以上、意思決定の高度化・定量化 |

> **注**: agent-blueprint.md では email-reader / email-drafter が Phase 1、property-scout が Phase 2 に分類されているが、本プロジェクトの実運用上の優先度（機会損失防止の緊急性）を踏まえ、property-scout / deal-analyzer を Phase 1 に据えている。

## 6. Executive プラットフォーム依存

本プロジェクトは以下の Executive プラットフォーム機能に依存する。

| 依存機能 | 用途 | 詳細 |
|---------|------|------|
| dashboard_* ツール群 | タスク管理・進捗可視化 | `dashboard_add_task`、`dashboard_update_task`、`dashboard_add_blocker`、`dashboard_add_artifact`、`dashboard_stats`、`dashboard_list` |
| handoff | 長時間セッションの引き継ぎ | 不動産投資判断は検討に長時間を要するため、セッション切れ対策として `/handoff` → `/resume` のフローを活用 |
| quality-gate | エージェント成果物の品質チェック | GATE 1（計画品質）＋ GATE 2（成果物品質）の保証フレームワーク |
| 依頼プロトコル | Executive への基盤変更依頼 | `[REQUEST_EXECUTIVE]` blocker による基盤変更依頼（global-architecture.md §6 参照） |

### 6.1 エスカレーションフロー

```
事業プロジェクト内でグローバルツール追加・基盤変更が必要と判断
  │
  ├─ dashboard_add_blocker(id=<task>, blocker="[REQUEST_EXECUTIVE] <内容>")
  ├─ dashboard_update_task(status="blocked")
  │
  ▼
dispatcher がブロッカーを検知 → ユーザー確認 → Executive 側で対応
```

## 7. 制約・リスク

| # | 制約・リスク | 内容 | 対応方針 |
|---|-------------|------|---------|
| 1 | 法的判断の最終責任 | 契約書レビュー・税務判断の最終責任は人間（弁護士・税理士）にある | AI エージェント出力は「助言」として扱い、最終決定は必ず人間が行う |
| 2 | 投資判断の最終決定 | エージェントの分析結果は意思決定の材料であり、投資判断の最終責任は投資家（ユーザー）にある | deal-analyzer の出力は収支計算の自動化に留め、Go/No-Go 判断は人間が下す |
| 3 | 遠隔運用の限界 | 札幌の保有物件における現地対応（緊急修繕・入居者トラブル等）は AI では代替不能 | 管理会社との連携体制を維持し、人間による現地対応フローを別途確保する |
| 4 | データ鮮度 | ポータルサイト・競売情報は更新タイミングが媒体によって異なる | property-scout の巡回頻度を適切に設定し、機会損失を最小化する |
| 5 | 事業特化エージェントの未整備 | Phase 1 のエージェントは未実装であり、構想段階である | agent-blueprint.md に基づき段階的に実装。各 Phase の完了基準を明確化する |

## 8. 運用ガイドライン

### 8.1 依頼時の request_group

ダッシュボード上で本プロジェクトのタスクをグループ化するために、以下の `request_group` を使用する。

| request_group | 用途 |
|--------------|------|
| `realestate-agent-design` | エージェント設計（agent-blueprint.md の策定・更新） |
| `realestate-phase1` | Phase 1 エージェント（property-scout、deal-analyzer）の実装 |
| `realestate-phase2` | Phase 2 エージェントの実装 |
| `realestate-phase3` | Phase 3 エージェントの実装 |
| `realestate-phase4` | Phase 4 エージェントの実装 |
| `realestate-docs` | 要件定義書・設計文書の作成・更新 |

### 8.2 成果物管理

- 全エージェント定義ファイルは `C:\dev\realestate\.opencode\agents\` に配置する
- 事業固有の設計文書は `C:\dev\realestate\docs\` に格納する
- 成果物は `dashboard_add_artifact` でタスクに紐付け、進捗を可視化する
